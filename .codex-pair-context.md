# Project context for codex-pair review

This is the **ask-llm** monorepo — an MCP (Model Context Protocol) server that bridges Claude with Google Gemini, OpenAI Codex, and local Ollama. Core idea: "Gemini/Codex/Ollama read, Claude edits" — leverage other LLMs' large context windows for analysis while Claude handles interaction and code editing.

Forked from [jamubc/gemini-mcp-tool](https://github.com/jamubc/gemini-mcp-tool) (unmaintained); now maintained at [Lykhoyda/ask-llm](https://github.com/Lykhoyda/ask-llm).

## Architecture you should know

Yarn workspaces monorepo with these packages:

- **`@ask-llm/shared`** (private) — registry, logger, commandExecutor, changeMode, serverFactory, ResponseCache
- **`ask-gemini-mcp`** (npm) — Gemini CLI wrapper + MCP server
- **`ask-codex-mcp`** (npm) — Codex CLI wrapper + MCP server. `packages/codex-mcp/src/utils/codexExecutor.ts` is the canonical source of truth for codex CLI invocation.
- **`ask-ollama-mcp`** (npm) — Local Ollama HTTP executor + MCP server
- **`ask-llm-mcp`** (npm, orchestrator) — Auto-detects installed providers and routes
- **`@ask-llm/plugin`** (Claude Code plugin) — Agents, skills, hooks. Distributed via marketplace.json at repo root.

## Critical invariants (must NOT be violated)

1. **Zero workspace imports in `packages/claude-plugin/scripts/codex-pair-watch.mjs`** — the hook ships via `git-subdir` marketplace install which has no `node_modules`. Any `import` from `@ask-llm/*` or `ask-*-mcp` throws `ERR_MODULE_NOT_FOUND` at startup. Use only Node built-ins (`node:fs/promises`, `node:fs`, `node:child_process`, `node:path`, `node:os`, `node:crypto`, `node:url`). See ADR-078.

2. **The codex-pair hook always exits 0** — `main().catch(...)` wraps everything. Every `process.exit()` is `process.exit(0)`. The hook must never break Claude's tool flow.

3. **LOW concerns stay in `.codex-pair-log.jsonl` only by default** — only HIGH and MED surface via `systemMessage` (ADR-077 threshold-in-hook design). The `surfaceThreshold: low` frontmatter option (shipping in v0.6.0 item #5) is the only sanctioned opt-up.

4. **codex-pair hook stays synchronous-blocking** — do not detach codex into a background process. The agent-accountability argument is load-bearing: Claude must see findings before yielding control to the user, otherwise the pair-programmer contract silently breaks on agentic work.

5. **Version bumps go through changesets** — never hand-edit version strings in `package.json`, `plugin.json`, `marketplace.json`. Use `yarn changeset` + `yarn changeset:version` (which auto-runs `scripts/sync-plugin-manifests.mjs`).

6. **codex CLI invocation in the hook mirrors `codexExecutor.ts`** — structural tests pin the flag list and stdin pattern. If you propose changes to the spawn contract, both files must be updated; otherwise the structural test fails CI.

## Things you SHOULD flag if you see them

- New `import` from a workspace package inside `codex-pair-watch.mjs`, `pre-commit-review.sh`, or any other script under `packages/claude-plugin/scripts/`.
- `process.exit(N)` where `N !== 0` anywhere in `codex-pair-watch.mjs`.
- `throw` outside the `main().catch` wrapper in the hook.
- Async background spawning patterns (e.g. `child.unref()`, `nohup`, fire-and-forget) in the hook.
- `concerns.low.map(...)` outside a `surfaceThreshold === "low"` gate.
- New CLI flags in `buildCodexArgs` that don't also exist in `codexExecutor.ts:buildArgs`.
- Shell-interpreted subprocess invocations (the variant that takes a single command string) when the safer `spawn` / `execFile` form would suffice.
- Hand-edited version numbers in any of the three plugin manifests.
- Test count regression below the 152-test baseline established by Phase 1.

## Things you should NOT complain about

- **Apparent duplication between `codex-pair-watch.mjs` and `codexExecutor.ts`** — the mirroring is intentional per ADR-078; structural tests guard against drift. Suggesting "DRY this out via a shared module" would re-introduce the `ERR_MODULE_NOT_FOUND` bug we just fixed.
- **Use of `readFileSync` at module load in `codex-pair-watch.mjs`** — module-load is one-shot, sync is fine there; don't suggest async migration for startup config reads.
- **"You should use a YAML parser library"** — the hand-rolled frontmatter parser in item #5 is required by the zero-workspace-imports constraint; pulling in `yaml` or `js-yaml` would defeat the marketplace install model.
- **"Why not Jest"** — we use Vitest. Not a regression.
- **English vs en-dash dashes in docs** — stylistic; not a real concern.
