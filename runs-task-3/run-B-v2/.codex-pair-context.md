# Project context for codex-pair (task 3)

This is a TypeScript library implementing **RFC 6902 JSON Patch** (which uses RFC 6901 JSON Pointer for path resolution).

**Stated requirements:**
- `applyPatch(doc, ops)` function: takes a JSON document and a list of patch operations, returns a new document with the patch applied
- 6 operations supported: `add`, `remove`, `replace`, `move`, `copy`, `test`
- TypeScript strict mode, Zod for op-shape validation
- Vitest tests covering happy paths and edge cases
- `tsc --noEmit` and `vitest run` must pass

**Critical RFC compliance points worth checking:**

1. **JSON Pointer escape sequences (RFC 6901 ¬ß3 + ¬ß4)**: The path `~0` represents the literal `~`, and `~1` represents the literal `/`. So path `/a~1b` resolves to key `a/b`, NOT keys `a`, `~1b`, or `a~1b`. Implementations that just split on `/` without un-escaping are incorrect.

2. **`test` op semantics (RFC 6902 ¬ß4.6)**: "The 'test' operation tests that a value at the target location is equal to a specified value." Equality is structural/deep equality ‚Äî `{a:1} === {a:1}` would (correctly) return false in JavaScript, but the `test` op should return true for these. Naive `===` or `JSON.stringify` comparisons are wrong for non-trivial values.

3. **Atomicity (RFC 6902 ¬ß5)**: "Operations are applied sequentially in the order they appear in the array. Each operation in the sequence is applied to the target document; the resulting document becomes the target of the next operation. Evaluation continues until all operations are successfully applied or until an error condition is encountered." On error, the implementation MUST not leave the document in a partially-modified state visible to the caller. Naive impls that mutate the input doc in place violate this ‚Äî when op #3 of 5 fails, the caller's doc has ops 1 and 2 already applied.

4. **Array index `-` (RFC 6902 ¬ß4.1, ¬ß4.2)**: The token `-` references the (nonexistent) element after the last array element. Valid for `add` (means "append"); invalid for `remove`, `replace`, `move from`, `copy from`, `test` (those need an existing index). Naive impls using `parseInt("-", 10)` get `NaN` and behave unpredictably.

5. **Invalid path/op behavior (RFC 6902 ¬ß4.x)**:
   - `remove` on a path that doesn't exist MUST error (¬ß4.2)
   - `replace` on a path that doesn't exist MUST error (¬ß4.3)
   - `move` where the `from` location is a prefix of the `path` location MUST error (¬ß4.4) ‚Äî can't move a subtree into itself
   - Unknown operation types MUST error

**Threat surface to check even if not in the spec:**
- Prototype pollution via crafted paths like `/__proto__/polluted` ‚Äî should be defended against in any production impl
- Input mutation in general ‚Äî function must return a NEW doc, not mutate the caller's

A correctness-focused review should evaluate the code against each of these specific RFC requirements, not just "does it look reasonable."
